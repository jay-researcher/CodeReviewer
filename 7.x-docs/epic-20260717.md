## 1. Epic 目标

本 Epic 在现有 Code Review 报告能力上，建立完整、可追溯、按角色控制的线上 Issue Review 工作流：

```text
Run Review
  → Report Preview / Discussion
  → Problems Handling
  → Auditor / Manager Review
  → Re-scan
  → Snapshot
  → Manual Pass
  → Sprint / Release Gate
```

重点解决：

1. Developer、Auditor/Leader、Manager 按角色和 responsible 范围访问功能。
2. Critical/High 问题通过“处理 → Re-scan → 再处理”形成闭环。
3. 同一个 Jira Issue 跨多个 Sprint 再处理时，保留历史关联，同时只审核本轮增量。
4. frontend/backend 普通代码 MR 分开出具问题清单；Company Config、SCR 延后到 GIT_VERSION Release Gate。
5. Issue Description、正式模板 Comment、报告、处理结果、审批和 Pass 均生成可审计快照。
6. Report Preview、Discussion、Problems、History 的职责清晰，不维护两套相互冲突的 Handling 数据。
7. Sprint Review 能区分 Batch Preview 和 Final Sprint Review，并提供发布门禁视图。

## 4. 页面与职责重组

### 4.1 Report Preview

Report Preview 定位为报告证据阅读入口，只保留：

- Preview、Compare、Raw、Download；
- Open Issue Review；
- Discussion；
- AI Assist。

移除正式 Handling、Approval、Re-scan、Manual Pass 和 Teams Delivery。点击 `Open Issue Review` 时，深链到对应 Issue、Review Cycle、Run 和 Finding。

### 4.2 Issue Review

Issue Review 是唯一的正式处理与审计入口，建议使用以下 Tab：

1. `Overview`：Issue/Sprint/Cycle 状态、完成率、阻碍和负责人。
2. `Problems`：三种处理、审批、Re-scan、Manager Exception、Submit。
3. `Discussion`：关联 Issue/Cycle/Run/Finding 的沟通记录。
4. `History & Snapshots`：Review Run、Description、Handling、Pass 快照。
5. `Pending Jira`：待创建 Jira 草稿。

Problems 提供子筛选：`All`、`Unhandled`、`Awaiting Approval`、`Re-scan Required`、`Resolved`。

### 4.3 Discussion

Report Preview 和 Issue Review 复用同一 Discussion 组件和 SQLite 数据源，不再维护 thread JSON 与 SQLite 两套正式数据。

Discussion 记录上下文：

```text
jira_key + cycle_id + run_id + finding_id（finding_id 可为空）
```

Report Preview 的 Discuss 页面 Reply 区块在视觉结构上与 Issue Review 右侧面板对齐。AI Chat 重命名或定位为 `AI Assist`，只提供解释、汇总和建议。

## 5. 核心数据模型

### 5.1 层级

```text
Jira Issue
  └─ Review Cycle（一次 Sprint 处理周期）
       ├─ Review Run Group（一次逻辑 Review）
       │    ├─ frontend Review Run
       │    └─ backend Review Run
       ├─ Deferred Release Resources
       ├─ Findings / Handling / Approval
       ├─ Discussions
       ├─ Jira Description Snapshots
       └─ Review / Pass Snapshots
```

### 5.2 Review Cycle

`review_cycles` 至少保存：

- `cycle_id`、`jira_key`；
- `sprint_id`、`sprint_name`、Sprint state；
- `cycle_started_at`、`cycle_closed_at`；
- Jira 状态回退和状态流转信息；
- `review_mode`：`issue`、`batch-preview`、`final-sprint`；
- 当前 Description Snapshot；
- 当前 MR 选择范围；
- 当前 Pass、Release Gate 状态。

同一个 Jira Issue 仍只显示一张主卡片；跨 Sprint 的多次处理展示为多个 Review Cycle。

### 5.3 Review Run Group

一次逻辑 Review 可能因 `config.yml` 中 `type=frontend|backend` 产生两份报告。必须使用父级 Run Group 关联，避免被误认为两次业务 Review。

`review_runs` 增加：

- `cycle_id`、`run_group_id`；
- `project_type`；
- `responsible_scope`；
- MR fingerprint 和 stable fingerprint。

### 5.4 Finding 身份

Finding 继续使用稳定指纹关联多个 Run。显示序号只用于 UI，不能作为持久主键。

## 6. Jira Issue 跨 Sprint 再处理

### 6.1 业务流程

Issue 第一次交付后，如再次发现问题：

```text
Dev Change Reviewed
  → Backlog
  → 加入新 Sprint（旧 Complete Sprint 仍保留）
  → Open
  → In Progress
  → Development Done
  → Run Review / Handling / Re-scan / Pass
  → Dev Change Reviewed
```

在本轮处理中，使用与 Issue Description 相同的正式模板在 Jira Comment 中说明继续修复的内容。

### 6.2 Sprint 识别

不能继续只取 Jira Sprint 列表第一项。系统必须：

- 保存全部 Sprint membership；
- 区分 Complete、Active、Future；
- 结合 Jira changelog、Sprint 加入时间和状态回退确定当前 Cycle；
- 将旧 Complete Sprint 保留在历史中；
- 即使当前 Sprint 后续关闭，仍可通过已保存 Cycle 找到当时报告和 MR。

### 6.3 本轮 MR 选择

MR 使用以下身份关联 Cycle：

```text
GitLab project + MR IID + head SHA
```

选择规则：

1. MR 必须关联当前 Jira Key。
2. 来源包括 Issue 字段/正式模板 Comment、Jira Remote Link、Development Panel、GitLab Issue Key 搜索和受控分支搜索。
3. 已在前一 Cycle 审核且 `head SHA` 未变化的 MR 自动排除。
4. 同一 MR 的 `head SHA` 变化时，作为新 revision 处理。
5. 提供本轮 MR 清单预览；Auditor/Manager 可审计地 Include/Exclude。
6. 历史发现不能把第一次已交付的旧 MR diff 再放进第二轮 Prompt。

### 6.4 增量 Review 边界

| 内容 | 用途 | 是否为本轮审核对象 |
| --- | --- | --- |
| 本轮普通 MR 的 `base_sha → head_sha` diff | 本轮代码增量 | Yes |
| 目标分支当前最新代码 | 影响、依赖和兼容性分析 | Context only |
| 本轮正式模板 Comment | 本轮修复目标 | Yes |
| 原始 Description/以前 Comments | 历史业务背景 | Context only |
| 以前 Cycle 的 MR diff | 已交付代码 | No |

原则：**审核对象是本轮增量，评估基线是目标分支当前最新代码。**

Prompt 分为：

```text
Current Review Scope
  - 当前 Sprint / Review Cycle
  - 本轮正式模板 Comment
  - 本轮普通 MR 清单与 incremental diffs

Current Target Context
  - 目标分支最新相关文件
  - 调用、依赖、配置及兼容性上下文

Historical Requirement Context
  - 原始 Description
  - 以前正式模板 Comments 的结构化摘要
  - 不包含以前 MR diff
```

## 7. MR 类型路由与 Deferred Release Resources

### 7.1 普通代码 MR

- `frontend` 与 `backend` 按 `config.yml` 项目类型分组；
- 分别生成问题清单和报告；
- 同属一个 Review Cycle 和 Run Group；
- LLM 只接收本轮该类型的增量 diff。

### 7.2 Company Config 与 SCR

Company Config、SCR 不是忽略项，而是 `Deferred Release Resource`：

- 关联当前 Jira、Sprint、Cycle；
- 不进入普通 Issue 增量代码 Prompt；
- 延后到 GIT_VERSION Release Gate；
- 核验配置、脚本、版本锁、资源引用、顺序、幂等、回滚和交付完整性。

Deferred 资源身份：

```text
Jira Key + Sprint ID + Cycle ID
+ GitLab Project + MR IID + Head SHA
```

如果本轮只有 Company Config/SCR，不得显示 `No MR` 或 Review Failed，应显示：

```text
Code MRs: 0
Deferred resources: 2
Issue Review: No code changes to review
Release Gate: Pending
```

状态需要区分：

- `Code Review Complete`；
- `Deferred Release Gate Pending`；
- `Release Ready`。

GIT_VERSION Review 只消费当前发布范围内尚未核验的 deferred resources。以前 Sprint 已完成核验的资源不重复审核；Head SHA 改变则按新 revision 重新核验。

### 7.3 Web 原生 Release Gate

正常发布工作流以 Web 为主，不要求 Manager 手动执行 CLI：

```text
Sprint Review Job
  → frontend/backend 报告与 deferred 摘要
  → Continue to Release Gate
  → GIT_VERSION MR 确认
  → Web Release Gate Job
  → READY / BLOCKED
  → 在线报告与发布决策
```

Web 设计要求：

- Run Review 中提供独立、明确的 `GIT_VERSION Release Gate` 卡片，不把 MR URL 混入 Jira/Sprint 普通输入。
- 只有 Manager 可见且可调用；API 同时校验权限。
- Sprint Job 发现一个 GIT_VERSION MR 时直接预填；发现多个时提供候选列表；未发现时允许手动输入。
- 继承 Sprint 上下文，但 Release Gate 始终是独立 Job，避免把发布级 diff 合并进普通 Sprint Prompt。
- 使用统一 Progress、Pause、Stop、Retry、Report Preview 和 History 能力。
- Job 卡片展示 Gate status、锁定源码仓库数、构建资源数和 blocker 数；最终状态合并确定性错误与配置范围内的 Critical/High Finding。
- 非 GIT_VERSION MR 在进入 LLM 前拒绝。
- CLI `python review.py --mr-url <GIT_VERSION MR>` 仅保留为自动化及故障排查入口。

7.0.5 已完成 Web Job/API、Manager 权限、GIT_VERSION 类型预检、Sprint Job handoff、Release Gate 结果卡片和在线报告入口。Review Cycle deferred MR 与锁定 build commit 的逐条持久化对账仍按本 Epic 后续 Phase 实现。

## 8. Jira Description 与 Comment 模板

### 8.1 Effective Description

完整有效 Issue Description 由以下内容按时间顺序组成：

1. Jira 原始 Issue Description；
2. 所有符合正式 Description 模板的 Comments。

普通讨论 Comment 不合并。

### 8.2 模板要求

模板必须包含三列表格：

| 中文 | English |
| --- | --- |
| 截图 | Screenshot |
| 说明 | Description |
| 补充 | Additional remarks |

第二列内容：

- 非缺陷：需求描述、需求分析、解决方案、预期结果；
- English：Requirement Description、Requirement Analysis、Proposed Solution、Expected Result；
- 缺陷：问题描述、问题分析、解决方法；
- English：Bug Description、Bug Analysis、Workaround。

第三列内容：

- 受影响的项目或功能范围 / Affected Project or Functional Scope；
- 涉及的文件清单 / Involved File Lists。

### 8.3 Description Snapshot

展示层级：

```text
Jira Issue
├─ Original Issue Description
├─ Cycle 1 / Sprint A
│  └─ 当时的 Effective Description
├─ Cycle 2 / Sprint B
│  ├─ Current Follow-up Comment
│  ├─ Effective Description
│  └─ Changes from previous cycle
└─ Cycle N
```

提供视图：

- `Original`；
- `Current Follow-up`；
- `Effective Description`；
- `Changes`。

快照保存：

- Jira Key、Sprint、Cycle；
- Description 或 Comment ID；
- ADF JSON 原文、安全渲染内容和纯文本索引；
- 作者、创建/更新时间；
- 模板语言和 Bug/非缺陷类型；
- 附件 ID、文件名、MIME、SHA-256；
- Jira 状态、关联普通 MR 和 Deferred MR；
- 快照版本、原因和采集时间。

截图等附件缓存到独立快照目录，SQLite 只保存索引与哈希。Comment 后续编辑时创建新版本，不覆盖旧快照。

快照时机：

1. Review Cycle 创建或首次 Run Review；
2. Description/正式模板 Comment 被编辑；
3. 本轮全部 Finding 完成处理；
4. Auditor/Manager Pass；
5. Release Gate 完成。

历史数据无法还原当时原文时，使用当前 Jira 内容回填，并标记 `Backfilled snapshot`。

## 9. Problems Handling 闭环

### 9.1 三种处理方式

1. `已整改，Pass 通过`（fixed）
   - 处理说明必填；
   - Critical/High 进入 `Re-scan Required`；
   - 后续 Run 中稳定指纹消失后才验证通过。
2. `不是阻碍，另报 Jira`（follow-up）
   - 处理说明、Issue Summary、Issue Description 必填；
   - 保存到 Pending Jira；
   - 默认不解除 Critical/High；
   - Manager 可填写原因并执行 Exception。
3. `不是问题，Pass 通过`（not-issue）
   - 理由必填；
   - Developer 提交后进入 `Awaiting Approval`；
   - Auditor/Manager 审批后才生效。

### 9.2 Pass 门禁

可配置阻碍严重级别，默认：

```yaml
app:
  review_workflow:
    blocking_severities: [Critical, High]
    require_rescan_for_fixed: true
    follow_up_unblocks_blocking: false
    manager_override_enabled: true
```

Manual Pass 前必须满足：

- 当前 Cycle 所有纳入范围的 Finding 都已有处理；
- Fixed Critical/High 已通过后续 Re-scan；
- Not Issue 已审批；
- Follow-up 阻碍已解决或具有 Manager Exception；
- 没有进行中的 Review Job；
- 最新 Run 之后没有未纳入的新 MR revision；
- Deferred 状态被明确展示，不与 `Release Ready` 混淆。

Handling 达到 100% 不等于 Pass；完成率、待审批、Re-scan 阻碍和 Deferred 门禁分别展示。

## 10. Issues Review History 与 Sprint Overview

### 10.1 Issue 卡片

在 Issue Summary 下方卡片区域展示：

- Critical、High 等级总数；
- 每个等级的未处理/已处理比值；
- 百分比和进度氛围设计；
- Remaining blockers；
- Manager exceptions；
- 当前 Cycle、最新 Run、更新时间和 responsible。

数字可点击并跳转到 Problems 对应级别的第一项。

### 10.2 多次处理历史

同一 Jira Issue 只显示一条主记录；多次加入 Sprint 形成多个 Review Cycle：

```text
Issue → Sprint → Review Cycle → Run Group → Application Runs → project type
```

Application 是报告、处理和 Release Gate 的业务边界；frontend/backend 是技术元数据。iTrade Client、Services Terminal、WVAdmin 分别作为独立 frontend 应用；iTrade Client 在 7.5 主版本下按 7.5.0.x/7.5.1.x 保留交付版本线，DPS 按 DPS9/DPS11 主版本线保留交付范围。History 展示创建时间、最后更新时间、处理方式数量、待处理数、快照、Pass 和 Release Gate 状态。

### 10.3 Sprint Overview

新增 Overview Tab，按 Jira Issue 所属 Sprint 聚合：

- Issue 总数、未生成报告、生成中、已生成；
- Handling、Re-scan Required、Ready for Pass、Passed；
- Critical/High 总数、已处理/未处理；
- Pending Jira、Manager Exception；
- Deferred Release Resources；
- Batch Preview / Final Sprint Review；
- Auditor responsible 范围和 Manager 全局范围。

## 11. Problems 页面体验要求

1. Problem 除标题外显示问题详情摘要，默认一行、最多两行，可展开完整内容。
2. `Submit handling` 简写为 `Submit`，按钮放在卡片右上角。
3. `No file · Persisting` 改为明确状态文案：
   - `Architecture / No specific file`；
   - `New in this run`；
   - `Still present since Run N`；
   - `Resolved after re-scan`。
4. 标题、区块和内容必须保留统一内边距，不允许贴边。
5. Issue Summary 使用可自动换行的两行输入组件，至少适合输入约 50 个中文字符。
6. 左右两列底部水平对齐。
7. Follow-up 布局采用清晰左右结构：左侧处理结果和处理说明，右侧 Issue Summary、Issue Description 摘要和 `Edit issue`。
8. Issue Description 通过独立弹窗编辑；卡片只回显摘要，不在主页面直接编辑 JSON。
9. Pending Jira 编辑器弹窗需有明确标题、说明、保存按钮、关闭按钮和认证错误状态，避免无意义控件。
10. Problems 右侧 Tab 必须可点击、可键盘访问，并正确维护 active/focus 状态。

## 12. ADF 编辑器

第一阶段支持：

- Paragraph、Heading；
- Bullet list、Ordered list；
- Table；
- Panel；
- Expand；
- nestedExpand 兼容性验证；
- Media/Image/Screenshot；
- Edit、Preview。

要求：

- 使用拖放和可视化组件编辑，不能要求用户直接编辑 ADF JSON；
- Expand 内支持表格、有序/无序列表和图片；
- 验证 Panel/表格内嵌内容及 nestedExpand 的兼容和降级策略；
- 保存前执行 ADF schema 校验；
- 编辑与 Preview 往返不得丢失节点；
- 本期保存 Pending Jira 草稿，后续由 JiraReviewer 创建真实 Jira Issue。

## 13. 全局表单与交互规范

所有页面统一：

- 必填字段标签使用红色 `*`；
- 字段级错误、表单级错误和服务端错误使用统一样式；
- 验证失败后聚焦第一个错误，并提供 `aria-describedby`；
- 提交期间按钮 disabled，并显示 `aria-busy`/loading；
- 防重复策略采用约 350ms 入口防抖、leading single-flight、幂等键；
- 不使用会延迟用户提交的纯 trailing debounce；
- 成功后给出一致的 toast/inline confirmation；
- 列表、Tab、弹窗和按钮支持键盘访问与可见 focus。

## 14. Run Review 与 Sprint 预检

### 14.1 Sprint 搜索

Sprint 字段由普通输入框改为搜索选择框：

- Focus 时展示最近一个月 Sprint；
- 格式：`e-Channel Sprint 1.4.75 (10085)`；
- 支持 ID 或名称搜索；
- 运行前验证 Sprint 存在且用户有访问权限；
- 保留清晰的 loading、empty、error 状态。

### 14.2 Review 模式

点击 Run Review 前加载 Sprint Issues 并检查状态：

- 存在非 `Development Done` Issue：显示确认对话框；继续则标记为 `Batch Issue Preview`。
- 全部为 `Development Done`：标记为 `Final Sprint Review`，用于 release readiness 检查。

Final Sprint Review 是 Sprint 工作流门禁，不替代明确通过 GIT_VERSION MR URL 运行的最终 Release Gate。

### 14.3 Progress

- 默认自动滚动到最新消息；
- 用户拖动滚动条或滚轮离开底部后暂停自动滚动；
- 提供 `Jump to latest`；
- 用户返回底部后恢复自动滚动；
- 同时存在多个 Job 时，根据 viewport 提供 `Maximize` / `Restore`；
- 最大化不能丢失其他 Job 状态；
- 页面重载后仍可恢复 Job 与日志状态。

## 15. Review Snapshot

当前 Cycle 的全部 Finding 已提交处理后生成不可变审核快照；后续审批、Exception、Re-scan 或 Pass 发生实质变化时生成新 revision，不覆盖旧版本。

快照包含：

- Issue、Sprint、Cycle、Run Group；
- frontend/backend 报告和 MR fingerprint；
- Description Snapshot；
- Finding 列表、三种处理结果、理由和审批；
- Re-scan 验证；
- Pending Jira 草稿；
- Deferred Resources；
- Discussion 引用；
- Pass readiness、Manager Exception、最终操作人和时间。

## 16. 数据迁移与兼容
